# LocalSkill Android

LocalSkill is a Kotlin Android application for connecting job seekers with verified local companies while giving administrators tools to manage verification, users, jobs, reports, and platform quality.

## Roles

- **Job Seeker**: discovers jobs, saves jobs, applies, manages profile/resume, tracks applications, and receives application/report notifications.
- **Company**: completes verification, manages company profile/documents, publishes jobs after approval, reviews applicants, schedules interviews, and receives recruitment/moderation notifications.
- **Administrator**: reviews company verification, moderates jobs and reports, manages users/categories, and receives verification/report notifications.

## Features

- MVVM + Repository Pattern with `StateFlow` and lifecycle-aware Compose collection.
- Firebase Authentication, Realtime Database, Storage, and client-side Firebase Messaging foundation.
- Job seeker discovery, applications, saved jobs, profile, resume, education, experience, and skills.
- Company verification, job lifecycle, applicant review, interview scheduling, and dashboard.
- Administrator dashboard, company/user management, job moderation, categories, reports, and activity log.
- Typed in-app notifications with deterministic event IDs and role-safe destination mapping.

## Technology Stack

- Kotlin, Jetpack Compose, Material 3, Navigation Compose
- Coroutines, `StateFlow`, one-time event channels
- Firebase Authentication, Realtime Database, Storage, Messaging
- DataStore Preferences, Coil, Activity Result APIs
- JUnit and Compose UI tests

## Architecture

The app intentionally keeps a pragmatic single-module MVVM structure:

```text
app/src/main/java/com/example/localskill/
├── model/       # Firebase-compatible app models and enums
├── repo/        # Repository interfaces and Firebase-backed implementations
├── services/    # Android/Firebase services and validation services
├── utils/       # Result wrappers, formatters, constants, error mapping
├── view/        # Compose screens, components, navigation, theme
└── viewmodel/   # Lifecycle ViewModels exposing immutable StateFlow
```

Firebase references are created in `AppContainer` and repository implementations, not composables or ViewModels.

## Firebase Services

Enable/configure these Firebase services for the Android package `com.example.localskill`:

- Authentication with email/password
- Realtime Database
- Firebase Storage
- Firebase Cloud Messaging
- App Check (debug provider for development, Play Integrity for release)

Deploy rules with Firebase CLI after review:

```sh
firebase deploy --only database
firebase deploy --only storage
```

## Firebase Messaging Scope

This Android app implements:

- FCM dependency and `FirebaseMessagingService`
- token refresh handling
- per-user app-scoped device token registration under `userDevices/{userId}/{deviceId}`
- notification channels
- typed payload validation
- safe app-launch pending intents

Outbound remote push delivery requires a trusted backend or Firebase Cloud Function. Do **not** embed FCM server keys, service-account JSON, Admin SDK credentials, or private Cloud Function secrets in the Android app.

In-app database notifications work independently of remote push delivery.

## App Check

Development should use Firebase App Check debug provider. Release should use Play Integrity provider and enforce App Check in Firebase Console for Realtime Database, Storage, and Cloud Functions if/when push-sending functions are added. Do not commit debug App Check tokens.

## Release Notes

- `minSdk` remains 31.
- Release minification is currently disabled until `assembleRelease` and regression testing validate R8/ProGuard behavior.
- Signing credentials must be supplied outside Git.
- Review `docs/release-checklist.md` before merging to `main` or tagging a release.

## Known Limitations

- Push sending is not implemented in this repository; a trusted backend/Cloud Function is required.
- Storage rules cannot fully validate company ownership across Realtime Database without trusted custom claims such as `companyId`; repository validation and path restrictions are used until backend claims exist.
- Instrumented tests require emulator/device availability.
